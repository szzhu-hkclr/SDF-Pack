## PHASE 0 — REPOSITORY SCAN

Before writing any deliverable, execute a full codebase survey:

1. **Language Census** — Identify every programming language present.
   Count lines of code (LOC) per language; compute LOC%.
2. **Dependency Manifest Audit** — Parse `package.json`, `requirements.txt`,
   `setup.py`, `pyproject.toml`, `CMakeLists.txt`, `Cargo.toml`, etc.
   Flag each dependency as `used_in_source: true|false` by grepping imports
   in actual source files.
3. **Source‑File Index** — Build a map of every source file with a 1‑line
   summary of its responsibility (module, class, or script purpose).
4. **Architecture Sniff** — Identify the top‑level architecture pattern
   (monolith, ROS2 multi‑node, microservice, mono‑repo, etc.),
   key design decisions, and explicit tradeoffs found in comments / docs.
5. **Metrics Harvest** — Scan for any quantifiable data:
   latency numbers, FPS, success rates, reduction percentages, test counts,
   benchmark results, config constants (control gains, frequencies, etc.).
   Mark each as `[measured]` or `[estimated]` with derivation notes.

> **GATE**: Do NOT proceed to Phase 1 until Phase 0 data is complete.

### 1.2 Rules
# T1-T8, L0-L4, when a scope or skill_name not included, add to proper track and/or levels, and warn in the very begining of all the wiki generation

| -- | --------------------------------------------------------------------------------- |
| R1 | Every `taxonomy_ref`  exist in `.devin/taxonomy.yaml`.                        |

| R3 | Do NOT list a technology that only appears in a manifest without source usage.     |

```yaml

page_meta:
  title: "Knowledge Base — English"
  language: en

  pattern: "<e.g., ROS2 multi‑node, modular monolith>"

      tradeoff: "<what was sacrificed>"

  - id: "<repo_title>-001"
    xyz_text: >
      <Action verb> <accomplished X> measured by <Y metric>
      by <doing Z technical method>.
    action_verb: "<verb>"
    metrics:
      quantified: "<plain English metric sentence>"
      values:
        - name: "<metric name>"
          value: "<number + unit>"
          source: "[measured] | [estimated]"
          derivation: "<how you got this number>"
    skills_referenced:
      - "T<track_num>-L<level_num>-<skill_name>"
    ats_keywords:
      - "<keyword>"
    interview_weight: <0.0–1.0>
    star_format:
      situation: "<context & background>"
      task: "<what needed to be done>"
      action: "<detailed technical actions taken>"
      result: "<outcomes with metrics>"
    source_files:
      - file: "<path>"
        lines: "<range>"
        relevance: "<why this file matters>"
```

### 3.2 Rules

| #  | Rule                                                                              |
| -- | --------------------------------------------------------------------------------- |
| K1 | Generate **5–8 bullets**, sorted by `interview_weight` descending.                |
| K2 | `xyz_text` ≤ 2 lines, action‑verb‑first, must contain ≥1 quantified metric.      |
| K3 | `star_format` is verbose interview prep — detail is encouraged.                   |
| K4 | Metrics must be derivable from code. Mark `[estimated]` if not exact.             |
| K5 | `interview_weight` higher for: strong metrics + deep technical + robotics relevance.|
| K6 | Every bullet MUST reference ≥1 `taxonomy_ref` from `skills_referenced`.           |
| K7 | `action_verb` must be a **strong** verb: Engineered, Architected, Optimized, etc. |
| K8 | Each bullet MUST include ≥1 `source_files` entry with line ranges.                |
